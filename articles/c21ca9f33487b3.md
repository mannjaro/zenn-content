## 概要

- ターゲット: NodejsFunctionを雰囲気で利用しているユーザ（私）
- ゴール: NodejsFunctionについてちょっとだけ詳しくなる

## 動機

CDKのNodejsFunctionを使ってデプロイする際、ローカルではうまくいくのに、GitHub ActionsなどのCI/CD環境下で動かなかったので気になった

**パターン1(成功する)**

- binやlibと同階層に実装がある
- ルートのpackage.jsonに依存関係が書かれている

```
.
├── backend
│   ├── package.json
│   └── src
│       └── index.ts
├── bin
│   └── cdk.ts
├── lib
│   └── cdk-stack.ts
└── package.json    <-- backendの依存関係を追加
```

**パターン5（成功する）**

https://tmokmss.hatenablog.com/entry/20220509/1652103822

- 下記のようにNodejsFunctionを実装する
  - `depsLockFilePath`を指定する
  - `bundling`オプションの`nodeModules`に必要なパッケージを明記する

```ts
new NodejsFunction(this, 'HonoFunction', {
  entry: path.join(__dirname, '../../backend/src/index.ts'),
  depsLockFilePath: path.join(__dirname, '../../backend/package-lock.json'),
  bundling: {
    nodeModules: ['hono']
  },
  handler: 'handler',
  runtime: lambda.Runtime.NODEJS_24_X
});
```

## なぜこうなるのか

[先ほどの記事](https://tmokmss.hatenablog.com/entry/20220509/1652103822)を参考に、aws-cdkのソースコードを読んでみる

https://github.com/aws/aws-cdk

ローカル実行時の流れ

NodejsFunction → Bundling.bundle → getLocalBundlingProvider → createLocalCommand → createBundlingCommand → exec(localCommand)

:::details コード抜粋

**NodejsFunction**

```ts
/**
 * A Node.js Lambda function bundled using esbuild
 */
export class NodejsFunction extends lambda.Function {
  constructor(scope: Construct, id: string, props: NodejsFunctionProps = {}) {
    // 中略
    {
      super(scope, id, {
        ...props,
        runtime,
        code: Bundling.bundle(scope, {
          ...props.bundling ?? {},
          entry,
          runtime,
          architecture,
          depsLockFilePath,
          projectRoot,
        }),
        handler: handler.indexOf('.') !== -1 ? `${handler}` : `index.${handler}`,
      });
    }
  }
}
```

**Bundling.bundle**一部抜粋

```ts
/**
 * Bundling with esbuild
 */
export class Bundling implements cdk.BundlingOptions {
  /**
   * esbuild bundled Lambda asset code
   */
  public static bundle(scope: IConstruct, options: BundlingProps): AssetCode {
    return Code.fromAsset(options.projectRoot, {
      assetHash: options.assetHash,
      assetHashType: options.assetHash ? cdk.AssetHashType.CUSTOM : cdk.AssetHashType.OUTPUT,
      bundling: new Bundling(scope, options),
    });
  }
  // 中略
  constructor(scope: IConstruct, private readonly props: BundlingProps) {
    this.packageManager = PackageManager.fromLockFile(props.depsLockFilePath, props.logLevel);
    // 中略
    const bundlingCommand = this.createBundlingCommand(scope, {
      inputDir: cdk.AssetStaging.BUNDLING_INPUT_DIR,
      outputDir: cdk.AssetStaging.BUNDLING_OUTPUT_DIR,
      esbuildRunner: 'esbuild', // esbuild is installed globally in the docker image
      tscRunner: 'tsc', // tsc is installed globally in the docker image
      osPlatform: 'linux', // linux docker image
    });
    this.command = props.command ?? ['bash', '-c', bundlingCommand];
    // 中略
    if (!props.forceDockerBundling) { // only if Docker is not forced
      this.local = this.getLocalBundlingProvider(scope);
    }
  }

  private createBundlingCommand(scope: IConstruct, options: BundlingCommandOptions): string {
    const pathJoin = osPathJoin(options.osPlatform);
    
    const esbuildCommand: string[] = [
      options.esbuildRunner,
      '--bundle', `"${relativeEntryPath}"`,
      // 中略
    ];
    return chain([
      ...this.props.commandHooks?.beforeBundling(options.inputDir, options.outputDir) ?? [],
      tscCommand,
      esbuildCommand.join(' '),
      ...(this.props.nodeModules && this.props.commandHooks?.beforeInstall(options.inputDir, options.outputDir)) ?? [],
      depsCommand,
      ...this.props.commandHooks?.afterBundling(options.inputDir, options.outputDir) ?? [],
    ]);
  }

  private getLocalBundlingProvider(scope: IConstruct): cdk.ILocalBundling {
    const osPlatform = os.platform();
    const createLocalCommand = (outputDir: string, esbuild: PackageInstallation, tsc?: PackageInstallation) => this.createBundlingCommand(scope, {
      inputDir: this.projectRoot,
      outputDir,
      esbuildRunner: esbuild.isLocal ? this.packageManager.runBinCommand('esbuild') : 'esbuild',
      tscRunner: tsc && (tsc.isLocal ? this.packageManager.runBinCommand('tsc') : 'tsc'),
      osPlatform,
    });
    const environment = this.props.environment ?? {};
    const cwd = this.projectRoot;

    return {
      tryBundle(outputDir: string) {
        // 中略
        const localCommand = createLocalCommand(outputDir, Bundling.esbuildInstallation, Bundling.tscInstallation);
        exec(
          osPlatform === 'win32' ? 'cmd' : 'bash',
          [
            osPlatform === 'win32' ? '/c' : '-c',
            localCommand,
          ],
          {
            env: { ...process.env, ...environment },
            stdio: [ // show output
              'ignore', // ignore stdio
              process.stderr, // redirect stdout to stderr
              'inherit', // inherit stderr
            ],
            cwd,
            windowsVerbatimArguments: osPlatform === 'win32',
          });
        return true;
      },
    };
  }
}
```

:::

`localCommand`の中身は以下の通り

```ts
    return chain([
      ...this.props.commandHooks?.beforeBundling(options.inputDir, options.outputDir) ?? [],
      tscCommand,
      esbuildCommand.join(' '),
      ...(this.props.nodeModules && this.props.commandHooks?.beforeInstall(options.inputDir, options.outputDir)) ?? [],
      depsCommand,
      ...this.props.commandHooks?.afterBundling(options.inputDir, options.outputDir) ?? [],
    ]);
```

確かに、特に`node_modules`を生成せずに`esbuild`を実行している
また、`beforeBundling`が`esbuild`前に実行されるので、inputDir上で`npm ci`が実行され`node_modules`が生成される

(ほぼ参考にした記事と同じことを書いているだけですね...)

## まとめと感想

エラーの原因は至極単純で、`esbuild`時に`node_modules`が存在しないというだけの話だった
packageのscriptsやActions上で`cdk deploy`前に一通り`npm ci`を実行するようにしようと思う

CDKは色々便利すぎて魔法に感じることが多かったので、ちゃんと向き合うべきと感じた
